Aprovação de código

O ideal é que todo o código desenvolvido, siga o padrão exigido, e que todo desenvolvedor entenda quais são os padrões e a importância deles, porém, quando uma área de código possui um heatmap (frequência de alterações ao longo do tempo) muito baixo, uma refatoração durante CR para impor padrões pode acabar sendo muito custosa de modo que o tempo investido refatorando aquilo, poderia ser melhor investido refatorando uma outra área com maior heatmap ou a própria entrega de valor para o cliente

  • Lógica simplificada
  • Código limpo
  • Desuso de condicionais (if, else)
  • DRY

Code Review por call

Para solicitar e realizar uma Code Review via chamada, é necessário seguir alguns fatores listados logo abaixo

  • Deve ser verificado disponibilidade na agenda do revisor
  • Ideal o revisor realizar duas Code Reviews no máximo por dia
  • Cada Code Review deverá ter no máximo uma hora
  • Dúvidas podem ser levantadas antes mesmo do código de produção se finalizado

Code Review por chat

Para solicitar e realizar uma Code Review via chat, é necessário seguir alguns fatores listados logo abaixo

  • MRs podem ser encaminhadas diretamente ao revisor
  • Feedback sobre o código pode ser enviado diretamente pelo gitlab, demarcando linhas e abrindo uma discussão
  • Códigos menores são mais fáceis, rápidas e objetivas de revisar
